RPA 구축 - RPA 구축
 
1. 프로젝트 개용 및 목적
- 목적 : 사내 RPA 시스템 및 제조 설비에서 발생하는 정형(excel, csv) 데이터를 안정적으로 통합 수집하고 이를 사내 AI 에이전트 (llm) 가 보안 규칙을 준수하며 쿼리 할 수 있도록 'Data-to-AI 통로(tool_api)' 를 구축한다.

- 핵심 제약사항 
 A.폐쇄망(air-gapped) 환경 : 외부 인터넷 및 퍼블릭 클라우드(AWS,GCP)연동 없이 사내 인프라에서 100% 독립적으로 동작해야한다.

B. 비동기 및 유연성 : 대용량 파일 유입시 타임 아웃을 방지하는 비동기 처리 구조를 갖춰야한다.

2. 목표 아키텍처 및 기술 스택 
명시한 아래 기술 스택은 목표 달성을 위한 '권장 표준안' 이다 제안 업체는 폐쇄망 구동 100% 오픈소스(무료 라이센스) 라는 핵심 전제 조건만 만족한다면 더 효율적이나 운영 안정성이 뛰어난 대안 기술 스택이 있을 겨이우 명확한 사유와 함께 자유롭게 변경하여 제안 할 수 있다. 
권장 스택 예시
- 수집 및 서빙 API : fastapi (python) , pydantic(타입/스키마 검증)
- 비동기 큐 & 워커 : redis + celery ( 또는 동급의 경량 비동기 프레임 워크)
- 데이터 레이크 포맷 : apache iceberg, pyiceberg
- 메타 데이터 카탈로그 : postgresql 
- 오브젝트 스토리지 : minIO( 비정형 파일 원본 및 parquet 저장용 ) 
- 초고속 쿼리 엔진 : duckDB( in-memory 기반 파티션 조회 ) 
- 벡터 검색 : Milvus

3. 핵심 기능 요구 사항 
REQ-01. 비동기 기반 원천 데이터 수집 API(Ingestion)
- RPA 서버 및 외부 시스템이 주기적(시간/일단위) 정형 데이터를 전송 할 수 있는 post 기반 rest api 를 제공한다. 
- 대용량 파일 수신시 네트워크 타임 아웃을 방지하기 위해 비동기 큐-워커 패턴을 적용한다.
- 수집 API는 파일을 받자 마자 즉시 응답(202 Accpted)를 반환하여 실제 데이터 파싱 및 스토리지 적재는 백그라운드 워커가 순차적으로 처리한다.
REQ-02. AI 에이전트 연동용 서빙 API 
본 시스템은 AI 에이전트(LLM) 가 데이터 베이스에 직접 SQL을 실행 하거나 파일 시스템에 접근하는 것을 엄격히 금지하면 오직 pydantic 기반으로 엄격하게 정의된 'Tool(도구) API'를 통해서만 데이터를 제공한다.

1. 스키마 탐색 API ( GET /agent/tools/schema)
- 에이전트가 현재 데이터베이스에 존재하는 모든 정형 컬럼명과 메타데이터 필드를 동적으로 파악할 수 있도록 Tool 명세(OpenAPI/JSON schema형식)를 반환한다. (RPA연동으로 신규 컬러이 추가되면 자동 갱신됨 )
2. 에이전트 전용 데이터 쿼리 tool ( POST /agent/tools/query)
- 에이전트는 자여니어를 바탕으로시스템이 정의한 규칙( 예. {"production_date":"2026-05-29","line_id":"FAB-1"} 에 맞춰 인자값을 던져 호출한다.
- 내부 쿼리 엔진(DuckDB등)은 이 인자값을 받아 파티션 기반으로 초고속 필터링을 수행한다.
- 결과값은 정형 데이터의 요약, llm의 context windows 를 낭비하지 않도록 최적화된 json 형식으로 반환 되어야한다.
